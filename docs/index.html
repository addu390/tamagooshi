    Quickstart

    Everything runs in the desktop simulator first, so you don't need a board to try it. You'll need PlatformIO and a C toolchain (the simulator builds with SDL + LVGL).

    Run the simulator offline

    Product defaults, no board or live data required.

    make sim                # product defaults
make sim BRAND=demo     # pick a brand

    Run against live data

    Start the dev stack, then point the simulator at its broker.

    make up                 # dev stack: broker + hub
make sim-live BRAND=demo
    Tip: add DEV=marc to bake a developer name into the build's branding.

    Configuration

    A brand is one config.yaml under brands/<id>/, in three groups: brand (identity), device (what ships in firmware), and hub (metrics and rules). Build a brand's headers with make brand BRAND=<id>.

    Build one below. Pick the identity, packs, and rules you want and the config.yaml updates live, ready to copy or download into brands/<id>/.

    Anything you leave out is simply off. A brand only bakes in the packs, games, and rules it lists, keeping the image small.

    Build & flash

    Happy with the config above? Flash it straight from the browser over USB. This writes the gooshi image, which compiles in every game, app, pack, theme, and typeface, then writes your selections as a small config partition so the same firmware boots as your brand. Add wifi under Transports and the flasher switches to the BLE + Wi-Fi image. Otherwise it flashes the BLE base.

    Build a lean image locally

    The browser image carries everything and picks what to show at runtime. For the smallest possible image, build only what a brand lists and skip the config partition entirely.

    make brand BRAND=<id>                              # generate headers into firmware/.gen/current
cd firmware && pio run -e m5stickc-plus -t upload    # build + flash only that brand (pick your board's env)

    A lean build compiles just its listed content, so nothing is filtered at runtime. Pass DEV=<id> to bake in a developer name.

    Metric sources

    A source is anything the hub polls for numbers. Each brand lists its sources under hub.sources in config.yaml, and the hub turns every reading into a metric the rules can act on. Sources are pluggable behind one Source port, so adding a provider is a small adapter.

    TypeWhat it reads
    demoSynthetic values with drift and scripted incidents.
    datadogThe latest point of a Datadog timeseries query.
    posthogA HogQL query or a saved insight, by short_id.
    + yoursDrop a folder under hub/src/sources/

    Give each metric a raw provider query. For example, PostHog metrics take either a query (HogQL) or an insight short_id. Keys stay out of config: source names the environment variables it depends on, and the hub fails fast if one is missing.

    hub:
  sources:
    - type: datadog
      interval_secs: 30
      metrics:
        - {key: cpu, label: CPU, query: "avg:system.cpu.user{*}", fmt: "{v}%"}
    - type: posthog
      project_id: "12345"
      metrics:
        - {key: signups, label: SIGNUPS, query: "SELECT count() FROM events WHERE event = 'signup'", kind: star}
        - {key: active, label: ACTIVE, insight: mmtixzFF}

    Set DATADOG_API_KEY and DATADOG_APP_KEY for Datadog, and POSTHOG_API_KEY for PostHog. Point a source at another region or self-hosted instance with its site or host field.

    Each source is a folder under hub/src/sources/<name>/ with its own config.py and source.py that registers a SourceProvider. Adding a provider is a new folder, not a change to the core.

    Run the hub

    The hub is the Python process that reads your sources and feeds the device. With a flashed device, this is the whole setup: one command on the machine that holds your API keys, and it pairs over BLE. One pairing carries hub metrics and Claude together.

    pip install -e hub      # once, Python 3.10+
make hub                # pairs with your device over BLE

    Development

    Use the MQTT stack to run the simulator against a live hub without a board. The wire format is defined once per side, in hub/src/wire and firmware/lib/wire.

    make up                 # dev stack: broker + hub via docker compose
make sim-live BRAND=demo # simulator against that broker
make logs               # tail hub logs
make hub-test           # hub unit tests

    Environment

    Behavior is driven by the brand config plus a few build-time environment variables.

    VariablePurpose
    TAMA_BRANDWhich brand config to build (default gooshi).
    TAMA_DEVDeveloper name/persona baked into the build.
    TAMA_TRANSPORTThe single link:protocol channel the hub uses to reach the device, e.g. ble:gatt or wifi:mqtt.
    TAMA_TRANSPORTSThe set of link:protocol channels compiled into the firmware, e.g. ble:gatt or ble:gatt,wifi:mqtt. Overrides the brand's device.transports.
    TAMA_BROKERMQTT broker address (default localhost:1883).
    TAMA_DEVICE_IDDevice identity for MQTT routing.
    DATADOG_API_KEY / DATADOG_APP_KEYCredentials for a datadog source.
    POSTHOG_API_KEYPersonal API key for a posthog source.
